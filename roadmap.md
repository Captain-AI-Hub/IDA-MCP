# IDA-MCP Roadmap

## 目标重心

项目下一阶段从“稳定 MCP / Gateway / IDA 多实例能力”转向“提供可直接操作的 Web 审计入口”，并围绕 **A2A（Agent-to-Agent）自动审计** 建立一条可落地、可观测、可恢复的执行链路。

核心原则：

- **IDA-MCP 继续作为底层能力层**：提供 IDA 实例接入、MCP 工具、资源读取、生命周期控制。
- **Web Backend 作为编排层**：使用 `LangGraph + DeepAgents` 驱动审计流程、任务状态、流式事件和人工介入。
- **Web Frontend 作为交互层**：提供任务发起、运行观察、结果回放、证据导出。
- **分层清晰**：浏览器不直接访问 `/mcp`，统一由 Web Backend 对接 `ida-mcp`。

---

## 当前基线

当前仓库已经具备：

- 稳定的 MCP / HTTP proxy 入口
- 多实例 gateway 与实例选择能力
- 较完整的 IDA 分析、修改、建模、资源接口
- `open_in_ida`、lifecycle、gateway 基础能力

当前仍未具备：

- 独立 Web 应用目录
- 面向浏览器的前端界面
- A2A 审计任务编排后端
- 审计状态流、日志流、报告流

---

## 目标架构

目标目录结构：

```text
web/
├── frontend/                  # Next.js Web UI
│   ├── app/
│   ├── components/
│   └── lib/
├── backend/                   # FastAPI + LangGraph + DeepAgents
│   ├── app/
│   │   ├── api/
│   │   ├── agents/
│   │   ├── services/
│   │   └── schemas/
│   └── tests/
└── README.md
```

运行分层：

```text
Browser -> frontend(Next.js) -> backend(FastAPI) -> ida-mcp(/mcp, /internal) -> IDA instance
```

默认端口规划：

- `frontend`: `127.0.0.1:3000`
- `backend`: `127.0.0.1:8000`
- `ida-mcp`: `127.0.0.1:11338`

## 阶段规划

## P0：打基础（当前优先级最高）

### 1. 建立 Web 工程骨架

目标：先把未来功能的落点定住，避免继续把 Web/A2A 能力散落进现有 `ida_mcp/`。

工作项：

- 新建 `web/frontend` 与 `web/backend` 目录
- 明确 Next.js + FastAPI 的职责边界
- 为 agent、api、service、schema 建立基础结构
- 在 `web/README.md` 固化端口、依赖和运行方式

完成标准：

- 仓库内存在稳定的 Web 目录边界
- 后续前后端开发不再污染 `ida_mcp` 核心实现

### 2. 建立 Backend 最小服务面

目标：先让浏览器访问一个真正的 Web 入口，而不是直接碰 `/mcp`。

工作项：

- 提供 FastAPI 基础应用入口
- 提供 `/healthz`、`/api/status` 等基础状态接口
- 接入 `ida-mcp` gateway 健康检查与实例摘要
- 约定统一错误格式与响应 envelope

完成标准：

- 浏览器可通过 Web UI 间接看到 `ida-mcp` 在线状态
- Backend 可作为 Web 与 MCP 之间的稳定中间层

### 3. 建立 Frontend 最小控制台

目标：先提供可打开的页面和基础状态面板。

工作项：

- 初始化 Next.js 页面结构
- 首页展示 backend / gateway / selected instance 状态
- 预留任务面板、日志面板、结果面板入口
- 统一基础布局与后续组件目录约定

完成标准：

- 打开 `http://127.0.0.1:3000` 可看到可用界面
- 页面能显示后端状态与基础连接信息

---

## P1：A2A 自动审计 MVP

### 4. 建立 LangGraph 审计编排主链路

目标：用结构化 agent graph 替代脚本式串联调用。

工作项：

- 建立审计任务 state schema
- 定义 planner / executor / reviewer / reporter 等节点
- 接入 `ida-mcp` 工具调用层
- 支持单次任务从创建到结束的完整状态流转

完成标准：

- 可从 Web 发起一条完整的自动审计任务
- 审计流程具备明确的节点、状态和输出

### 5. 引入 DeepAgents 与子代理协作

目标：让多代理能力从“未来计划”进入真正可执行阶段。

工作项：

- 为不同审计职责定义 agent 角色
- 区分任务分解、证据收集、漏洞判断、报告汇总职责
- 支持子代理结果回传主图状态
- 建立失败重试、超时和人工接管点

完成标准：

- A2A 审计至少支持一个稳定可复现的场景
- 主代理与子代理职责边界清晰

### 6. 建立流式运行观察能力

目标：让 Web 能看到“任务正在做什么”，而不是只拿最终结果。

工作项：

- backend 向 frontend 输出 SSE 事件流
- 推送阶段状态、工具调用、关键日志、错误事件
- 区分用户可见日志与调试日志
- 为中断 / 恢复预留事件类型

完成标准：

- 用户可以实时观察任务推进过程
- 审计失败时可定位卡在哪个阶段

---

## P2：可恢复、可审查、可交付

### 7. 引入 checkpoint / resume / HITL

目标：让长任务可暂停、可恢复、可人工确认。

工作项：

- 为 LangGraph 引入持久化 checkpointer
- 基于 `thread_id` 管理审计会话
- 在高风险动作前加入人工确认点
- 支持恢复被中断的长任务

完成标准：

- 长任务不会因单次连接断开而完全丢失
- 关键操作可人工审查后继续

### 8. 建立审计报告与证据链

目标：从“有输出”升级到“输出可验证”。

工作项：

- 统一报告 schema
- 报告中关联函数、地址、伪代码、交叉引用、字符串等证据
- 支持导出 Markdown / JSON
- 区分结论、风险等级、证据、待确认项

完成标准：

- 审计结果可复查、可追溯、可二次加工

### 9. 建立可观测性与调试链路

目标：降低 A2A 系统接入后的排障成本。

工作项：

- 接入 LangSmith 或等价 tracing 能力
- 统一 backend / graph / ida-mcp 的 request correlation id
- 增加关键步骤的性能统计与失败归因
- 明确 Web 层与 IDA 层错误边界

完成标准：

- 能回答“哪一步失败了、为什么失败、是否能重试”

---

## P3：产品化收敛

### 10. 建立策略化审计模板

目标：把 A2A 审计从“通用对话”收敛为“可复用策略”。

工作项：

- 定义基础模板：样本初筛、字符串追踪、危险函数审计、控制流异常点扫描
- 支持按目标类型选择模板
- 支持模板级参数与报告模板

### 11. 加强权限边界与安全提示

目标：明确 unsafe 工具与高风险动作的界面和流程限制。

工作项：

- 对 `py_eval`、`dbg_*`、修改类工具做显式风险标记
- 为自动修改/调试操作增加确认机制
- 明确只读审计模式与可修改模式

### 12. 面向外部使用者整理开发/部署文档

目标：让 Web + A2A 能被独立部署和使用。

工作项：

- 完善 `web/README.md`
- 补充本地开发、生产部署、依赖安装、环境变量说明
- 固化最小 demo 与截图/流程说明

---

## 明确不做的方向

- 不让浏览器直接把 `/mcp` 当普通页面访问
- 不把 Web 页面直接嵌进 `ida_mcp` 核心服务
- 不以 `py_eval` 作为 A2A 主执行路径
- 不在没有状态管理与可观测性的前提下直接堆多代理功能
